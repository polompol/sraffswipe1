# Комиссия: как платит работодатель и все спорные моменты

Документ описывает полный цикл денег «заведение → StaffSwipe» и разбор всех
конфликтных сценариев с работодателем. Всё, что помечено ✅ — уже работает в
коде; 🔜 — включается после подключения ЮKassa (нужны ИП + договор с ЮKassa).

---

## 1. Что и когда начисляется ✅

- Комиссия = **10% от оплаты смены** (`COMMISSION_PCT`, настраивается),
  минимум за смену — `COMMISSION_MIN_RUB` (по умолчанию 0 = выключен).
- Оплата смены считается от ставки в вакансии: `perShift` — как есть;
  `perHour` — ставка × длительность (ночные смены через полночь учитываются).
- Начисляется **только за фактически закрытую смену** — когда обе стороны
  подтвердили выход (работник «я на смене» + заведение «человек пришёл»),
  либо когда оператор закрыл спор в пользу «смена была».
- **Одна запись на смену** (уникальный `match_id`) — двойное начисление
  невозможно даже при повторных запросах.
- Неявка, отмена, мэтч без смены — комиссии **нет**. Заведение платит только
  за результат. Это главный аргумент в продажах: «нет человека — нет денег».

## 2. Как заведение платит ✅

Два способа — работают одновременно, заведение выбирает удобный.

**Способ А. Баланс (аванс) — рекомендуемый.**
1. Заведение закидывает аванс: переводит оператору по СБП → оператор в
   админке жмёт «Баланс +N ₽» (после подключения ЮKassa — кнопки
   «+1000/3000/5000 ₽» в профиле, оплата картой, зачисление вебхуком).
2. Смена закрылась → комиссия **списывается с баланса автоматически**
   (атомарно, двойное списание невозможно), в чат смены падает отметка
   «Комиссия N ₽ списана с баланса ✓». Никаких счетов и напоминаний.
3. Баланс виден в профиле; каждое движение — в журнале (`wallet_txns`),
   так что вопрос «куда делись деньги» всегда разбирается по истории.
4. Денег на балансе не хватило → комиссия просто уходит в счёт (способ Б).

**Способ Б. Постоплата по счёту.**
1. Заведение видит свой счёт в профиле: карточка **«Комиссия сервиса · 10%»** —
   сумма, число смен, срок оплаты (`/billing/commission`).
2. Раз в неделю оператор открывает админку → блок «Комиссия к счёту» —
   список заведений с суммами (сортировка по убыванию долга).
3. Оператор выставляет счёт: СБП по номеру телефона / перевод по реквизитам /
   счёт для юрлица (вне приложения — мессенджер, почта).
4. Заведение платит → оператор жмёт **«Оплачено»** → все pending-начисления
   помечаются `paid`, заведению приходит уведомление в Telegram
   «Оплата получена, счёт закрыт ✓».

Юридическая основа — публичная оферта (`docs/legal/offer.md`): пользуясь
сервисом, заведение акцептует условие о комиссии за закрытую смену.

## 3. Что будет, если не платить ✅

- Срок оплаты — **7 дней** (`COMMISSION_DUE_DAYS`) с момента закрытия смены.
- Просрочка → **публикация новых вакансий блокируется** автоматически
  (402 с понятным текстом). Уже опубликованные смены и переписка работают —
  мы не рушим начатые договорённости, только перекрываем новую ценность.
- В профиле карточка счёта подсвечивается: «Счёт просрочен — публикация
  новых смен приостановлена до оплаты».
- Оператор отметил оплату → блок снимается **сам** (он вычисляется по данным,
  ничего вручную включать не нужно).
- Злостный неплатёж (недели, игнор) → оператор блокирует аккаунт целиком
  (механизм `blocked` уже есть) и, при желании, взыскивает по оферте.

Почему блок публикации, а не пени: на этом рынке заведению нужны люди
«на завтра». Невозможность повесить смену — мгновенная и достаточная боль;
пени в MVP не собрать, а конфликтность они повышают.

## 4. Спорные сценарии с работодателем — полный разбор

| # | Сценарий | Что происходит | Статус |
|---|----------|----------------|--------|
| 1 | «Человек не вышел», а работник отметился (код/гео) | Автоматически открывается **спор**, обе стороны уведомлены, решает оператор: «смена была» (комиссия начисляется) или «неявка» (бьёт по надёжности работника, комиссии нет) | ✅ |
| 2 | «Человек не вышел», работник и не отмечался | Фиксируется **неявка**, комиссии нет, спор не нужен | ✅ |
| 3 | Работодатель молчит: работник отметился, заведение не подтверждает | Смена висит в `confirmed`. Работник жмёт «Проблема» → спор → оператор. Правило разбора: код прихода знал только работодатель → раз код введён, работник был на месте → засчитать | ✅ (авто-закрытие по таймеру — 🔜 при масштабе) |
| 4 | «Смена была короче, человек ушёл раньше» | Комиссия считается от ставки в вакансии — **вакансия = договор**. Если стороны реально сократили смену, оператор в споре решает по факту; в MVP — вручную (пере-выставить сумму), системная корректировка — 🔜 | ✅/🔜 |
| 5 | «Не буду платить, я сам нашёл этого человека» (обход сервиса) | Кодом не решается. Ответ: комиссия начисляется только за смены, закрытые в приложении; ушли «мимо кассы» — заведение теряет надёжность, отзывы, ленту проверенных работников. Долг по уже закрытым сменам остаётся, при неоплате — блок | ✅ (блок) |
| 6 | «Оплатил, а блок не сняли» | Оператор жмёт «Оплачено» — блок снимается автоматически в ту же секунду. Если заведение оплатило, а оператор ещё не увидел — пишет в поддержку (кнопка в профиле) | ✅ |
| 7 | «Начислили дважды за одну смену» | Технически невозможно: уникальный `match_id` в таблице комиссий + идемпотентное начисление | ✅ |
| 8 | «Вы посчитали не ту сумму» | Сумма всегда = ставка из вакансии × длительность × 10%. Заведение само указало ставку при публикации; расчёт виден в карточке счёта. Ошибся в ставке → урок на будущее, оператор может простить разово («Оплачено» без денег) | ✅ |
| 9 | Заведение отменяет подтверждённую смену в последний момент | Комиссии нет (смена не закрыта), но по жалобе работника оператор выносит предупреждение (`warnings`); систематические отмены → блок. Компенсация работнику — 🔜 (с эскроу) | ✅ |
| 10 | Фейковое заведение / накрутка смен своим людям | Комиссию они начисляют сами себе — экономически бессмысленно. Накрутка рейтинга ловится оператором (отчёты, повторяющиеся пары «заведение-работник») | ✅ |
| 11 | Заведение оспаривает саму комиссию («не знал») | Оферта акцептуется при регистрации, процент показан в профиле постоянно, а не мелким шрифтом. Первому недовольному — разовое прощение и объяснение, рецидив — стандартные правила | ✅ |
| 12 | Переплата / двойной перевод | Вне приложения (MVP): оператор возвращает переводом или зачитывает в счёт следующей недели | ✅ (процесс) |

## 5. Что изменится с ЮKassa 🔜

1. **Автосписание**: заведение привязывает карту (сохранённый способ оплаты
   ЮKassa) → комиссия списывается автоматически в конце недели, чек по 54-ФЗ
   уходит сам. Ручной этап оператора исчезает.
2. **Оплата из приложения**: кнопка «Оплатить счёт» в карточке комиссии
   (инфраструктура платежей уже есть — `/billing/yookassa/payment`).
3. **Эскроу/предоплата смены** (по желанию заведения): деньги за смену
   замораживаются при подтверждении и выплачиваются работнику после закрытия.
   Это же — главное оружие против обхода сервиса (сценарий №5).

## 6. Правила для оператора (шпаргалка разбора споров)

- Код прихода введён верно → работник был на месте (код знал только
  работодатель). Засчитывать смену.
- Гео-отметка есть, кода нет, заведение отрицает → звонить обеим сторонам;
  по умолчанию доверие стороне с лучшей историей (надёжность/отзывы/стаж).
- Первый спор у заведения — решать мягко, объяснять правила.
- Повторные споры одного заведения по схеме «люди работают, платить не хочу» —
  предупреждение, затем блок. Смотреть паттерн в админке (жалобы по мэтчам).
- Любое решение фиксируется кнопками «Засчитать смену» / «Зафиксировать
  неявку» в админке — история остаётся в системных сообщениях чата смены.
